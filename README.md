### 2차 - 연결 유지

연결을 유지한 상태로 여러 메세지를 주고받고, 서로 연결을 종료시키는 방법에 대해 공부함
따라서 [[Server/Sample1.cs]]에서 연결을 유지한 상태로 최대 메세지 전송 횟수를 넘거나 연결이 끊길 때까지 메세지를 주고 받도록 변경했고,
[[Client/Sample1.cs]] 또한 동일하게 여러 메세지를 주고받을 수 있도록 함
하지만 클라이언트 측은 메세지의 수신과 송신을 따로 분리해야 원할한 통신이 가능하다는 것을 깨닫고 [[Client/Sample2.cs]] 제작을 시작함
위 파일에서 송신과 수신을 다른 메서드로 만들어 Task.WhenAny로 둘이 다른 타임라인에서 돌도록 분리했고,
두 Task가 연결 상태를 공유하기 위해 CancellationTokenSource에 대해 공부함

> - 또한 Codex로 문제를 찾으며 발견한 문제점이 너무 많아 이번엔 변경사항 작성도 Codex에게 맡김

[[Client/Sample2.cs]]와 [[Server/Sample2.cs]]를 점검하면서 여러 비동기 작업이 하나의 `Dictionary`와 `StreamWriter`에 동시에 접근할 수 있고, 연결 자원의 생성과 정리 위치도 명확하지 않다는 것을 알게 됨.

[[Server/Sample3.cs]]에서는 클라이언트 목록을 `ConcurrentDictionary`로 변경해 접속과 종료, 방송이 동시에 일어날 때 발생할 수 있는 동시성 문제를 보완함.

또한 각 `ClientInfo`가 bounded `Channel<string>` 형태의 송신 큐와 전용 송신 Task를 갖도록 변경함
다른 작업들은 `StreamWriter`에 직접 쓰지 않고 `SendAsync()`를 통해 메세지를 큐에 넣으며, 실제 송신은 전용 Task가 순서대로 처리하도록 만들었음.

`ClientInfo`가 `TcpClient`, `Stream`, reader, writer를 직접 소유하고 `IAsyncDisposable`을 구현하게 하여 연결 자원의 생성과 정리도 한곳에서 담당하도록 변경함.

서버에서는 명령어 처리를 `CommandManager()`로 분리하고 `/stop` 명령어를 추가함
서버 전체가 하나의 `CancellationTokenSource`를 공유하게 하여 `/stop`이 입력되면 accept, 송신, 수신 작업에 종료 신호가 전달되도록 함
접속한 클라이언트에게 자신의 번호도 알려주도록 보완함

[[Client/Sample3.cs]]에서는 연결 자원을 `ClientInfo`로 분리하고, 포트 범위와 빈 메세지 같은 사용자 입력을 검증하도록 변경함
송신과 수신 Task가 같은 취소 토큰을 공유하게 하고, 한쪽 작업이 끝나면 나머지 작업과 연결 자원도 함께 정리되도록 종료 흐름을 수정함.

이번 보완을 통해 여러 비동기 작업이 공유 자원에 접근할 때는 단순히 Task를 나누는 것뿐 아니라, 동시성 제어와 자원 소유권, 취소 및 종료 흐름까지 함께 설계해야 한다는 것을 공부함.
아직 세부적으로 개선할 수 있는 부분은 남아 있지만, 3차에서 확인한 주요 문제는 보완했으므로 여기서 3.5차를 마무리함. 이후 필요한 개선 사항은 4차 기능을 구현하고 실제 문제를 확인하면서 함께 수정할 예정.
